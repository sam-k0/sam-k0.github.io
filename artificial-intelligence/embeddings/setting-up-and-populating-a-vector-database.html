Setting up and populating a vector database

This note is gonna go over how to set up and populating a local vector database of choice, which is gonna be qdrant. I'd imagine the setup process for most other vectordb services like milvus is pretty similar.

Step 0: Gathering requirements: qdrant, embedding model & data

Getting a container running in docker is pretty trivial so I'm gonna skip this. All that needs to be done is running docker run -p 6333:6333 qdrant/qdrant, which will pull & run the official qdrant image and expose port 6333 to the host. This gives us access to the web-dashboard at
http://localhost:6333/dashboard.

The next step would be to choose an embedding model that is used to turn strings into vectors.
Some background on it is covered in The magic of word embeddings.
A pretty solid and resource-light model would be nomic-embed-text... At least it was the perfect balance of context window, embedding quality and hardware requirements back when it released 2 years ago. Filtering by "Embedding" on ollama.com also shows that it is still quite popular:

Anyways, after ollama pull nomic-embed-text-ing the embedding model, it is time to take a look at the most important matter, the data.

Models matter

Trying out different models will definitely give you different results every time, so I'd recommend testing with 2 or 3 different ones. I'd probably go with qwen3-embedding and embeddinggemma. If you want multilingual embeddings, try nomic-embed-text-v2-moe.

Back when I built a semantic search component for a university project, the class was given a subset of the wikidata knowledge graph as data source, but you can of course take any dataset you like.
So the next step now is to create a python script that pulls out the relevant data from the knowledge graph, embeds the texts, generates an ID for each embedding (qdrant requires this) and then pushes it to qdrant.

But first, install all the required python packages for doing this: pip install SPARQLWrapper qdrant-client requests. Of course, if you don't use knowledge graphs, omit SPARQLWrapperfrom the list.

Imports, constants and clients

Next is the python code to import packages and define some constants like the embedding api endpoint and SPARQL prefixes to make retrieval easier:

import requests
from SPARQLWrapper import SPARQLWrapper, JSON
from qdrant_client import QdrantClient
from qdrant_client.http import models
import hashlib

print("Starting embedding loader...")


OLLAMA_URL = "http://localhost:11434/api/embeddings" 
PREFIX = """
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX : <http://h-da.de/fbi/art/>
PREFIX owl: <http://www.w3.org/2002/07/owl#>
PREFIX xml: <http://www.w3.org/XML/1998/namespace>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
PREFIX wd: <http://www.wikidata.org/entity/>
BASE <http://h-da.de/fbi/art/>
"""

sparql = SPARQLWrapper("http://fuseki:3030/art-dataset/query")

# check if the db already exists
client = QdrantClient(url="http://qdrant:6333")
if client.collection_exists("artists"):
	print("Collection 'artists' already exists")
	exit(1)
	#client.delete_collection("artists")

collection_created = False
batch_size = 100
offset = 0
total_inserted = 0

sparql again is just related to the data loading and points to the SPARQL endpoint to retreive data from. The client holds the reference to the qdrant server. Depending on how your docker setup looks like, you will need to change this url to whatever you set up.

Anyways, after getting the client reference, the first thing I do is check if the "collection" (vector db term for a "table") called "artists" already exists. Because why would we want to embed all of the data again when starting up the program?

Lastly are some other variables I'll need later in the big loop where all of the data gets embedded and pushed.

The embedding function and ID generation

After the constants, I defined some global methods to get the embeddings from the model and to assign one piece of embedded information an id:

def get_embedding(text):
	r = requests.post(
		OLLAMA_URL, # the endpoint defined above
		json={"model": "nomic-embed-text", "prompt": text}, # the model
		timeout=30
	)
	r.raise_for_status()
	return r.json()["embedding"] # only return embedding

  

def make_id(uri):
	# Artist IDs are provided as URIs in Wikidata, so we hash it to get an integer ID
	return int(hashlib.md5(uri.encode()).hexdigest()[:8], 16)

You will probably need to adjust the make_id function for your data - or keep it if you wanna hash whatever data you provide.

The big loop

This is where all of the data gets processed by the steps mentioned earlier:

  1. Load a batch of data from knowledge graph using SPARQL
  2. Embed & generate IDs
  3. If the current batch is the first batch (the collection "artists" doesn't exist already), create a collection called "artists"
  4. Upsert the embedded data to qdrant

I put the whole loop-code below and gonna go over the code equivalents of the steps above. Keep in mind that around all of the code is one giant while True:.

Step 1: loading batched data from the datasource

sparql.setQuery(PREFIX + f"""
	SELECT ?artist ?label ?desc WHERE {{
		?artist a :person ;
		rdfs:label ?label .
		OPTIONAL {{ ?artist :description ?desc . }}
	}}
	LIMIT {batch_size}
	OFFSET {offset}
	""")

sparql.setReturnFormat(JSON)
results = sparql.query().convert()

artists = [
	{
		"id": r["artist"]["value"],
		"name": r["label"]["value"],
		"desc": r.get("desc", {}).get("value", "")
	}
	for r in results.get("results", {}).get("bindings", []) #type:ignore
]

if not artists:
	print(f"No more artists to fetch at offset {offset}")
	break

print(f"Fetched {len(artists)} artists (offset: {offset})")

As you can see above, there's a bunch of query magic happening to the SPARQL endpoint, but what it does is just fetch batches of data (id, name and description) for each artist and turns it into a list of dictionaries with the respective keys.

Step 2: embedding and generating IDs

vectors = []
payloads = []
ids = []

for artist in artists:
	text = f"{artist['name']}. {artist['desc']}" # prepare text for embedding
	emb = get_embedding(text)

	vectors.append(emb)
	ids.append(make_id(artist["id"]))
	payloads.append({
		"wikidataID": artist["id"],
		"name": artist["name"]
	})

print(f"Generated {len(vectors)} embeddings")

Iterating through the artists list from step 1, I concatenate the artists' name and description into one long string and pass it to the get_embedding function that requests the embedding model to turn that text into a vector. I then populate a list vectors with these results.
Alongside, I populate the lists ids, which contains the ids of the artists and payloads which holds additional information about the artist.

Important note regarding the lists

Please keep in mind that all of these three lists MUST be the same length, so keeping care of them alongside each other is VERY dangerous. I'd recommend packing all of the information into one container (like a dataclass?) and only keeping one list, which makes it easier to guarantee the same amount of data!

Step 3: Assuring the collection exists
Remember that variable collection_createdfrom the first code snippet at the top?
Yeah, that's gonna come into play now:

if not collection_created:
	vec_dims = len(vectors[0])		
	print(f"Vector dimension: {vec_dims}")
	
	client.create_collection(		
		collection_name="artists",		
		vectors_config=models.VectorParams(		
			size=vec_dims,
			distance=models.Distance.COSINE
		)
	)
	collection_created = True

If the collection already exists, we exit the code earlier, but now, if the collection doesn't exist AND we're inside the loop, we wanna take the first probe of vectors to use as configuration data for the collection.

The most important information to create a collection is the dimensions of the vectors. All vectors must have this exact size, for example a common size would be 768 dimensions.
So, I get the vec_dims and pass them to the config, next to the distance parameter, which distance function is used to determine the similarity of two vectors.
In this case, I opted for cosine similarity. Other functions include dot, euclid and manhattan, each with their own use cases' pros and cons.

The main reason to use cosine similarity is that I want similar artists to be in close proximity (in vector space) even though they may use different wordings in their description.

Step 4: Pushing data to qdrant
Almost done! The last step is just pushing data to the database:

client.upsert(
	collection_name="artists",
	points=models.Batch(
		ids=ids,
		vectors=vectors,
		payloads=payloads
	)
)

total_inserted += len(artists)
print(f"Inserted batch into Qdrant. Total: {total_inserted} artists\n")
offset += batch_size

Using the qdrant client upsert function, I push a batch of ids, vectors & payloads to the database. These are the same lists from step 2, and yes, they must be the same length.
Then finally, do some keeping track of offsets for batching and we're done!

You'll be able to look at your collection through the web dashboard!

For the sake of context, I put the complete pipeline code below:

import requests
from SPARQLWrapper import SPARQLWrapper, JSON
from qdrant_client import QdrantClient
from qdrant_client.http import models
import hashlib

OLLAMA_URL = "http://llm-service:11434/api/embeddings"
PREFIX = """
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX : <http://h-da.de/fbi/art/>
PREFIX owl: <http://www.w3.org/2002/07/owl#>
PREFIX xml: <http://www.w3.org/XML/1998/namespace>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
PREFIX wd: <http://www.wikidata.org/entity/>
BASE <http://h-da.de/fbi/art/>
"""
def get_embedding(text):
	r = requests.post(
		OLLAMA_URL,
		json={"model": "nomic-embed-text", "prompt": text},
		timeout=30
	)
	r.raise_for_status()
	return r.json()["embedding"]

  

def make_id(uri):
	return int(hashlib.md5(uri.encode()).hexdigest()[:8], 16)

sparql = SPARQLWrapper("http://fuseki:3030/art-dataset/query")
# check if the db already exists
client = QdrantClient(url="http://qdrant:6333")
if client.collection_exists("artists"):
	print("Collection 'artists' already exists")
	exit(1)
	#client.delete_collection("artists")

collection_created = False
batch_size = 10
offset = 0
total_inserted = 0

while True:
	# step 1
	sparql.setQuery(PREFIX + f"""
		SELECT ?artist ?label ?desc WHERE {{
			?artist a :person ;
			rdfs:label ?label .
			OPTIONAL {{ ?artist :description ?desc . }}
		}}
		LIMIT {batch_size}
		OFFSET {offset}
		""")
	
	sparql.setReturnFormat(JSON)
	results = sparql.query().convert()

	artists = [
		{
			"id": r["artist"]["value"],
			"name": r["label"]["value"],
			"desc": r.get("desc", {}).get("value", "")
		}
		for r in results.get("results", {}).get("bindings", []) #type:ignore
	]
	
	if not artists:
		print(f"No more artists to fetch at offset {offset}")
		break
	
	print(f"Fetched {len(artists)} artists (offset: {offset})")
	
	# embed current batch
	vectors = []
	payloads = []
	ids = []
	# step 2
	for artist in artists:
		text = f"{artist['name']}. {artist['desc']}" # prepare text for embedding
		emb = get_embedding(text)

		vectors.append(emb)
		ids.append(make_id(artist["id"]))
		payloads.append({
			"wikidataID": artist["id"],
			"name": artist["name"]
		})
	
	print(f"Generated {len(vectors)} embeddings")

	# step 3	
	# if its the first batch , we create a collection on qdrant server
	if not collection_created:
		collection_size = len(vectors[0])		
		print(f"Vector dimension: {collection_size}")
		
		client.create_collection(		
			collection_name="artists",		
			vectors_config=models.VectorParams(		
				size=collection_size,
				distance=models.Distance.COSINE
			)
		)

		collection_created = True
	
	#step 4
	# push batch to qdrant
	client.upsert(
		collection_name="artists",
		points=models.Batch(
			ids=ids,
			vectors=vectors,
			payloads=payloads
		)
	)
	
	total_inserted += len(artists)
	print(f"Inserted batch into Qdrant. Total: {total_inserted} artists\n")
	offset += batch_size
